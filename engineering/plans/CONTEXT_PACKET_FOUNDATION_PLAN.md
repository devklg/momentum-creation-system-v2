## 1. Purpose

The Context Packet Foundation defines the implementation boundary for `context_packet.v1` before production context assembly code is written.

Its purpose is to ensure that every runtime agent receives a governed, scoped, auditable packet assembled by the Context Manager, rather than retrieving knowledge directly from MongoDB, Neo4j, ChromaDB, GraphRAG, or ad hoc runtime queries.

The foundation gives Sprint 2 implementation a clear target:

- one packet schema shape,
- one assembly owner,
- one validation boundary,
- one provenance and citation strategy,
- one exclusion rule for candidate or review-only knowledge,
- one Team Magnificent scope model,
- one EN/ES language policy.

## 3. Proposed Schema: `context_packet.v1`

The implementation should align with the ratified runtime packet model while preserving a compact foundation contract for early implementation.

```ts
interface ContextPacketV1 {
  schemaVersion: "context_packet.v1";
  packetId: string;
  requestId: string;

  tenantId: string;
  teamId: string;
  baId?: string;

  agentId: "steve_success" | "michael_magnificent" | "ivory";
  objective: string;
  language: "en" | "es";

  userContext: UserSessionContext;
  approvedKnowledgeReferences: ApprovedKnowledgeReference[];
  graphContextReferences: GraphContextReference[];
  vectorContextReferences: VectorContextReference[];
  eventContextReferences: EventContextReference[];

  constraints: ContextConstraint[];
  excludedKnowledge: ExcludedKnowledgeReference[];
  provenance: ContextPacketProvenance;

  createdAt: string;
}
```

Required field notes:

- `packetId`: unique packet identifier generated by the Context Manager.
- `requestId`: caller-provided request identifier for traceability.
- `tenantId`: tenant scope for the packet.
- `teamId`: Team Magnificent team scope.
- `baId`: required when a Brand Ambassador session, BA-owned private context, relationship context, journal context, or training state is involved.
- `agentId`: runtime agent receiving the packet.
- `objective`: task-specific goal, such as success interview, training support, relationship coaching, invitation drafting, or session resume.
- `language`: `en` or `es`.
- `userContext`: current user/session context, including session id, mode, current state, and permitted user-owned context summary.
- `approvedKnowledgeReferences`: active approved knowledge included for agent use.
- `graphContextReferences`: graph-derived references selected by the Context Manager.
- `vectorContextReferences`: semantic/vector-derived references selected by the Context Manager.
- `eventContextReferences`: current or recent runtime events relevant to the session.
- `constraints`: runtime rules, privacy rules, compliance rules, language rules, and agent-specific boundaries.
- `excludedKnowledge`: knowledge intentionally withheld, with reason codes.
- `provenance`: retrieval and assembly audit trail.
- `createdAt`: ISO timestamp of packet creation.
- `schemaVersion`: always `context_packet.v1`.

## 4. Context Manager Rule

The Context Manager is the only assembler of context packets.

No agent, route, UI surface, tool worker, or workflow may construct a production `context_packet.v1` directly. Callers may request context, but only the Context Manager may:

- validate request scope;
- retrieve approved knowledge through approved retrieval interfaces;
- rank and budget context;
- exclude candidate/review-only knowledge;
- attach provenance;
- produce the final packet.

---

## 5. Agent Boundary Rule

Agents must not query MongoDB, Neo4j, ChromaDB, or GraphRAG directly.

Agents receive Context Packets and act from the packet. Direct retrieval by an agent is invalid even when the agent has an immediate need for more context. Missing context must produce either:

- a request for a refreshed packet;
- a degraded packet path;
- a clarifying question to the user.

---

## 6. Knowledge Lifecycle Rule

Candidate and review-only knowledge must be excluded from normal context packets unless explicitly authorized by a ratified review workflow.

The default packet must exclude:

- candidate knowledge;
- review-only knowledge;
- rejected knowledge;
- superseded knowledge unless included for historical explanation;
- archived knowledge;
- knowledge without source traceability;
- unapproved translations represented as approved knowledge.

---

## 7. Approved Knowledge Inclusion Rules

Approved knowledge may be included only when all of these are true:

- lifecycle status is active;
- governance status is approved or explicitly not required;
- source traceability exists;
- tenant and team scope match the request;
- agent domain is allowed for the current objective;
- language is same-language, human-reviewed translation, marked machine translation, or language-neutral fallback;
- privacy and relationship scope pass validation;
- inclusion fits the token/context budget.

Every included item must carry:

- knowledge id;
- source id;
- title or compact label;
- summary or excerpt;
- language;
- domain;
- inclusion reason code;
- retrieval source category: graph, vector, event, direct canonical lookup, or mandatory runtime rule.

---

## 8. Provenance and Citation Strategy

Each packet must preserve a compact provenance trail sufficient to answer:

- what was included;
- why it was included;
- what was excluded;
- why it was excluded;
- which retrieval layer selected it;
- which source record supports it;
- whether language fallback or translation was used;
- whether the packet is complete, degraded, or failed.

Agent-facing citations should reference packet-local source labels instead of exposing raw database details. Runtime audit records may keep full store-level ids.

Recommended provenance shape:

```ts
interface ContextPacketProvenance {
  generatedBy: "context_manager";
  retrievalStartedAt: string;
  retrievalCompletedAt: string;
  directPersistenceMode: {
    mongo: "direct";
    neo4j: "direct";
    chroma: "direct";
    gatewayFallbackAvailable: true;
  };
  includedRefs: Array<{ refId: string; sourceId: string; reasonCodes: string[] }>;
  excludedRefs: Array<{ sourceId: string; reason: string }>;
  degradedReasons?: string[];
}
```

## 9. Token and Context Optimization Strategy

The Context Manager should optimize for useful context, not maximum context.

Rules:

- mandatory runtime constraints always fit first;
- active task template comes before general background;
- same-session context beats distant history;
- approved knowledge beats broad narrative context;
- summaries are preferred over full transcripts;
- vector and graph retrieval return references plus compact summaries, not raw dumps;
- repeated references are deduplicated by canonical knowledge id;
- low-relevance context is excluded with a reason code;
- packet metadata records token estimate and compression status.

Initial budget guidance:

- constraints: required minimum, always included;
- active template/objective context: 1 current task frame;
- approved knowledge: 5 to 8 compact items;
- graph references: 3 to 5 references;
- vector references: 3 to 5 references;
- event references: recent and task-relevant only;
- user/session context: summarized current state, not full transcript.

---

## 10. Team Magnificent Scope Rules

Every normal packet is Team Magnificent scoped.

Required rules:

- `teamId` must resolve to Team Magnificent scope.
- Context must not become THREE-wide.
- THREE International remains the upstream authority for enrollment, genealogy, registration, and patronage.
- The app mirrors only the Team Magnificent slice.
- BA-owned private context must belong to the packet `baId`.
- Prospect/relationship context must be scoped to the active BA and authorized objective.
- `.com` prospect-facing compliance boundaries must be included when the objective could generate copy that later reaches prospects.

---

## 11. EN/ES Language Handling

Supported runtime languages are English and Spanish.

Language priority:

1. Same-language approved knowledge.
2. Human-reviewed translation.
3. Marked machine translation.
4. Language-neutral template.
5. Clarifying question or degraded packet.

Rules:

- `language` must be `en` or `es`.
- Machine translation must be marked and cannot be represented as approved bilingual source material.
- If the active task template is unavailable in the requested language, the packet must record fallback status.
- Agents must respond in the packet language unless a task-specific clarification requires otherwise.

---

## 12. Validation Strategy

Validation should happen at the Context Manager boundary before a packet is returned.

Validation must check:

- required fields are present;
- `schemaVersion` equals `context_packet.v1`;
- tenant/team/BA scope is consistent;
- agent id is valid;
- objective is allowed for the agent;
- language is supported;
- requested context scopes are allowed;
- candidate/review-only knowledge is excluded unless explicitly authorized;
- private context owner matches `baId`;
- relationship context is permission-scoped;
- provenance exists for every included item;
- Gateway fallback is still available but not the primary persistence path;
- packet status is complete, degraded, or failed.

Invalid packets must fail closed. Degraded packets must include safe agent instructions and missing-context reasons.

## 14. Risks and Mitigations

| Risk | Mitigation |
| --- | --- |
| Agents bypass the Context Manager for convenience | Enforce agent boundary in runtime interfaces and tests. |
| Candidate knowledge leaks into BA guidance | Default exclusion, lifecycle filters, and candidate-specific test cases. |
| Packets become transcript dumps | Token budget rules, summaries, dedupe, and compact references. |
| Provenance is too thin for audit | Require source ids, reason codes, retrieval layer, and exclusion reasons. |
| Language fallback is invisible | Require translation status and fallback flags in every packet. |
| Team scope broadens into THREE-wide claims | Validate Team Magnificent scope and include upstream authority constraints. |
| Gateway fallback is removed prematurely | Treat fallback removal as out of scope and not approved. |
| S1.5 drifts into Sprint 2 implementation | Acceptance criteria require planning-only output and no production code changes. |
