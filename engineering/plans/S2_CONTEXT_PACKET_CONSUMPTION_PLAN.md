## 1. Objective

Plan how Steve Success, Michael Magnificent, Ivory, and the Agent Runtime Orchestrator consume `context_packet.v1` without assembling context or retrieving directly from stores.

The Context Manager remains the only assembler of Context Packets.

## 3. Core Rule

Agents consume Context Packets only.

Agents and agent orchestration must not:

- query MongoDB directly;
- query Neo4j directly;
- query ChromaDB directly;
- call GraphRAG directly;
- import direct persistence adapters;
- import Gateway fallback clients;
- assemble Context Packets;
- include candidate/review-only knowledge by default.

## 4. Context Manager Boundary

Only Context Manager may:

- validate request scope;
- retrieve approved knowledge;
- select relationship context;
- select journal context;
- apply exclusion rules;
- apply token budget rules;
- produce provenance;
- assemble `context_packet.v1`;
- mark packet status as complete, degraded, or failed.

Agent Runtime may request a packet, validate a returned packet, and pass packet-local context to the agent.

## 5. Shared Packet Requirements

Every agent packet should include:

- `schemaVersion: "context_packet.v1"`;
- packet id;
- request id;
- tenant scope;
- Team Magnificent scope;
- authenticated `baId`;
- agent key;
- allowed objective;
- runtime mode;
- language;
- session id;
- current state;
- runtime rules;
- guardrails;
- approved knowledge references;
- exclusions;
- retrieval audit;
- packet status;
- generated-by metadata identifying Context Manager.

## 6. Agent-Specific Requirements

- Success Interview objective;

## 7. Packet Validation Before Agent Use

Agent Runtime should validate:

- schema version;
- agent key matches requested agent;
- objective is allowed for that agent;
- tenant and Team Magnificent scope are present;
- `baId` is present for BA-scoped sessions;
- packet generated by Context Manager;
- packet status is complete or degraded for substantive guidance;
- failed packet blocks substantive guidance;
- candidate/review-only exclusions are present when relevant;
- language is English or Spanish;
- provenance exists.

## 8. Missing, Degraded, And Failed Context

Missing required context:

- request refreshed packet;
- ask a clarifying question;
- use safe degraded response;
- stop if safety requires it.

Degraded packet:

- agent may respond only within the packet's safe degraded instructions;
- agent must not invent missing facts;
- response should identify that context is limited if user-facing copy requires it.

Failed packet:

- agent must not continue substantive guidance;
- surface safe fallback;
- capture non-persistent event envelope fact.

## 9. Packet-Local Source Use

Agents may cite or reference only packet-local labels, summaries, and approved source references.

Agents must not expose:

- raw Mongo ids;
- raw Neo4j queries;
- raw Chroma ids;
- GraphRAG implementation details;
- internal adapter names;
- Gateway fallback implementation details.

## 10. Static Boundary Checks

Future implementation approval should include static checks that fail if agent runtime files import:

- Mongoose models;
- MongoDB clients;
- Neo4j drivers or adapters;
- Chroma clients or adapters;
- GraphRAG clients;
- direct persistence adapter modules;
- Gateway fallback clients;
- ad hoc retrieval helpers.

Only approved Context Manager request interfaces may be used.

## 11. QA Expectations

Future implementation approval should require tests for:

- packet validation success;
- missing packet fields;
- invalid agent/objective pair;
- candidate/review-only exclusion;
- failed packet blocks guidance;
- degraded packet safe response;
- Context Manager-only assembly;
- no direct retrieval imports;
- EN/ES handling;
- packet-local source handling.

Mandatory gates remain:

- `pnpm typecheck`
- `pnpm build`
- `pnpm --filter @momentum/server test`
