# Executive summary

Standing rule. THREE International is the single source of truth and the final authority on all sponsorship, enrollment, placement, and compensation. Every record described in this document mirrors THREE’s records for Team Magnificent’s own operational visibility. The Team Magnificent app never disputes or overrides THREE. If at any point our records differ from THREE’s records, we update ours to match.
Team Magnificent is invitation-only. Entry to the .team site requires an access code, generated by Kevin from the admin dashboard and assigned to a sponsoring Brand Ambassador. The access code is the gate, and it also mirrors THREE International’s genealogy for Team Magnificent’s own operational visibility — every code traces back through the team to TM-01 (Kevin). The app has three client surfaces (admin, .team, .com), one server, three databases written through a single gateway, and a small set of external services (Telnyx for SMS and voice, an email provider, YouTube for the Dr. Dan video). # A.  The .team signup page

## A.1  Purpose
The signup page exists so a new BA who has just enrolled in THREE International under a Team Magnificent sponsor can create their account on teammagnificent.team. The page is the only public entry point to .team. Without a valid access code, no one can sign up. There is no public marketing entry to .team — access is exclusive.
## A.2  Where it lives
- Route: teammagnificent.team/register
- Linked from the .team login page (“Have an access code? Create your account.”)
- No SEO indexing — robots: noindex, nofollow.
## A.3  Page sections, top to bottom
### Header strip
- Team Magnificent wordmark in gold, top-left.
- No nav. The page does only one thing.
### Eyebrow + headline
- Eyebrow: “Team Magnificent · Brand Ambassador” in gold mono, uppercase, letter-spaced.
- Headline: “Create your account.” in Bebas Neue, very large.
- Sub-line: “You have been personally sponsored onto Team Magnificent. Enter the access code your sponsor gave you to create your account and begin your first 72 hours.”
### The form (single column, vertical stack)
- Access code — single text input, monospace font, auto-uppercase, placeholder “TM-XX”. Validated live: as the user types, the page hits /api/auth/verify-code in real time. When the code is valid, the field shows a green check and a line below reads “Sponsored by: [BA full name]”. When invalid, red border and “Code not recognized. Check with your sponsor.”
- First name — text input.
- Last name — text input.
- Email — email input. Validated for format. Required for login and password reset.
- Phone — tel input with formatting (auto-formats as the user types). Required — Michael calls this number to conduct the BA interview.
- THREE International username — text input. The username the BA uses to log into back-office at THREE.
- THREE International BA ID — text input. The numeric or alphanumeric BA ID assigned by THREE at enrollment.
- Password — password input, with strength indicator. Minimum 8 characters, at least one number and one letter.
- Confirm password — password input. Must match.
- Checkbox: “I agree to the Team Magnificent terms and to be contacted by my sponsor and by Team Magnificent communications.”
- Submit button: “Create my account” in gold. Disabled until the access code is verified and all required fields are filled.
### Footer
- Single line: “Already have an account? Sign in.”
- Compliance line below: “Team Magnificent is an independent operational team inside THREE International.”
## A.4  What happens when the form is submitted
On submit, the .team client posts the form payload to POST /api/auth/register. The server runs the following sequence in order. Any step that fails returns an error and aborts the rest — the BA record is not created unless every step passes.
- 1. Validate access code exists and is currently active.
- 2. Resolve the sponsor BA from the access code. This is the BA who owns the code; this becomes the new BA’s sponsor on Team Magnificent.
- 3. Validate email is not already in use by another BA on .team.
- 4. Validate THREE International BA ID is not already in use by another .team account.
- 5. Hash the password (bcrypt or argon2id).
- 6. Generate a new internal BA ID for Team Magnificent’s own records.
- 7. Triple-stack write: insert the new BA into MongoDB (bas collection), create a (:BA) node in Neo4j with a SPONSORED relationship to the sponsor BA, write an identity blob to ChromaDB for future semantic retrieval.
- 8. Issue a JWT and set it as an http-only cookie.
- 9. Trigger the welcome flow — enqueue Michael to call this number, send the welcome email.
- 10. Return success, redirect the client to /welcome.
## A.5  Validation rules in plain language
- Sponsor immutability — Once the access code resolves at step 2, that sponsor is locked in the new BA’s record. It cannot change later. Same principle as prospect sponsor immutability on .com, just applied to BA-to-BA.
- One-way email lookup — Email and THREE BA ID both must be unique per .team account.
- No silent partial records — If any step from 7 onward fails, the whole transaction rolls back. The new BA either fully exists or does not exist at all. No half-created accounts.
- Access code stays valid — After signup, the same code remains active. The next BA the same sponsor brings in uses the same code.
## A.6  What the user sees if something goes wrong
- Invalid access code: red border on the code field, “Code not recognized. Check with your sponsor.” Submit button stays disabled.
- Email already in use: red border on the email field, “An account already exists with this email. Sign in instead.” with a link to /login.
- THREE ID already in use: red border on the THREE BA ID field, “This THREE BA ID is already registered on Team Magnificent.”
- Password too weak: red border on the password field with the specific reason.
- Network or server error: a banner at the top of the form, “Something went wrong. Please try again, or contact your sponsor.”

Below is the diagram of the full BA cycle. Left lane is the .team side (the BA experience). Right lane is the .com side (the prospect experience). The center column traces the access code genealogy back to TM-01. The dashed gold curve along the left edge shows that the cycle repeats — every new BA eventually starts another cycle one level down.

## B.1  Phase 1 — Kevin controls the gate
Kevin generates an access code from the admin dashboard — the only place codes can be created. The code is assigned to a sponsor BA. From that point forward, the code is permanently tied to that BA. Paul holds TM-07 for life. Every new BA Paul sponsors uses TM-07.
## B.2  Phase 2 — The sponsor enrolls the new BA into THREE International (off-app)
Sponsorship into THREE itself happens entirely outside Team Magnificent’s app, using THREE’s own enrollment tools. This is what makes the new BA an actual THREE BA on the company’s books, in the company’s genealogy, in the company’s binary. After enrollment, the sponsor gives the new BA their access code by text or phone.
## B.3  Phase 3 — The new BA signs up on .team
The new BA goes to teammagnificent.team, follows the link to /register, and creates their .team account using the access code. The signup process (Section A above) records the sponsor binding in Team Magnificent’s own data — the operational record that mirrors THREE’s genealogy for our team’s use. The new BA then enters the welcome flow.
After welcome, Michael calls them and conducts the voice interview — captures why they’re doing the business, financial goals, time commitment, and anything useful for the sponsor to know. The transcript and scoring are fed back to the BA’s record and to their upline cockpit. After Michael, the new BA enters the Fast Start Guide for the first 72 hours: compensation plan training, binary placement, GLP-THREE plus other THREE products, building their initial prospect list, identifying their first two candidates. The 10-step new member orientation happens live with Kevin or Paul on Zoom, scheduled from inside .team.
## B.4  Phase 4 — The new BA invites their first prospect
After the orientation, the new BA opens the invitation generator inside .team. Per the rule established in the App Description Document: the BA speaks with the candidate first — SMS or scripted phone call, real human contact — then generates a personalized link from the tool. The link opens the presentation page (tm-prospect-glp3-v3-UPDATED.html), which leads with Dr. Dan’s 17-minute video, followed by market opportunity, product detail, the system explainer, the 2-in-72 visualization, and testimonials. When the prospect watches the video to completion, the video_complete event fires, the server places the prospect in the team-wide holding tank with a position number, and the prospect is shown the six-section dashboard. From the dashboard’s Section 6, they choose their next move: callback from their inviting BA, or seat at the next Tuesday 7pm Pacific live with Kevin and Paul.
## B.5  The cycle completes
When the prospect enrolls in THREE through their sponsoring BA, that BA requests their own access code from Kevin. Kevin generates the next TM-NN code from the admin dashboard, assigns it to the new BA, and Phase 1 begins again — one level down the tree. The same loop runs at every level, with every BA, forever.
The access code chain is a record of this. TM-01 is Kevin. TM-07 is Paul (in the example diagram). Each TM-NN traces back through the chain to TM-01. Each BA who holds a code carries our operational mirror of THREE’s sponsorship record. THREE International remains the final authority on the actual genealogy; the access code chain reflects it for Team Magnificent’s own visibility.

# C.  Technical system architecture

Below is the system architecture diagram. Three client surfaces sit at the top: the admin dashboard (Kevin only), the .team client (BA-facing), and the .com client (prospect-facing). All three call into a single application server, which writes through a single persistence gateway that fans every write to three databases simultaneously. External services (Telnyx, an email provider, YouTube’s IFrame API, and THREE International itself) sit at the bottom.

## C.1  Client surfaces
Three separate clients, identical stack, distinct purposes.
### Admin Dashboard — teammagnificent.team/admin
- Audience: Kevin only (eventually also Paul). Access-controlled at the route level — only specific BA IDs can reach the /admin tree.
- Generates access codes — the only place codes can be created.
- Views the entire team tree, both through the access-code genealogy and through THREE’s data when available.
- Sees pending callback requests, Michael interview transcripts, the full audit log.
### .team client — teammagnificent.team
- Audience: every Brand Ambassador on Team Magnificent.
- Public route: /register (the signup page).
- Authenticated routes: /welcome, /michael, /fast-start, /training, /orientation, /invitations.
- Login via email + password, returning a JWT in an http-only cookie.
### .com client — teammagnificent.com
- Audience: prospects who clicked an invitation link.
- Token-gated routes: /p/{token} resolves to the presentation page; after video_complete, the dashboard for that prospect.
- No login. The token in the URL is the identity.
### Stack details
- All three clients use the same stack: Vite + React 19 + TypeScript + Tailwind 3 + shadcn primitives.
- Brand tokens shared via a packages/shared workspace: gold (#C9A84C), teal (#2DD4BF), cream (#F5EFE6), on near-black (#0A0A0A).
- Typography: Bebas Neue for display, DM Sans for body, DM Mono for monospace.
## C.2  Application server
One Node + Express + TypeScript server, mounted at /api, hosts every endpoint the three clients call. It is divided into six domain modules:
- Auth — Access-code validation, signup, login, JWT issuance, password hashing, password reset flow.
- BA domain — Brand Ambassador profile, sponsor relationship (immutable after first set), THREE ID mapping, profile updates.
- Invitations — Invitation token mint and resolve, status tracking, /p/{token} resolution for the .com client.
- Placement — Holding tank logic, video_complete event handling, position assignment (monotonic, never reshuffled, team-wide pool).
- Realtime — Server-Sent Events stream for the dashboard’s live placement section. New prospects arriving cause an event to fan out to every connected dashboard.
- Notifications — Callback alert SMS to BAs (via Telnyx), welcome email, webinar reservation confirmation, audit logging.
## C.3  Persistence gateway and the triple stack
Every write that needs to persist hits the Universal Gateway at localhost:2525. The gateway fans the write to all three databases. This is the existing pattern from Kevin’s infrastructure, used by other projects already.
### MongoDB — the system of record
- access_codes — every code Kevin has generated, who it is assigned to, when, and its current state.
- bas — every Brand Ambassador profile: name, email, phone, THREE username and ID, password hash, sponsor BA ID, photo, etc.
- invitations — every invitation token a BA has generated, status, prospect data when known.
- prospects — the holding tank: every prospect placed, position number, sponsor BA ID, status, timestamps.
- callback_requests — every callback request from the .com dashboard, the prospect data, the intent radio choice.
- ba_alerts — every SMS dispatch to a BA, audited.
- michael_interviews — every transcript and scoring, linked to the BA record.
### Neo4j — the operational genealogy graph
Neo4j stores the relationships for Team Magnificent’s own operational queries. This is what makes our team’s genealogy mirror queryable in any direction — ancestors, descendants, distance to root. THREE International remains the source of truth; Neo4j gives us our own view.
- (:BA)-[:SPONSORED]->(:BA) — captures who sponsored whom across our team. Set at signup. If THREE’s record ever differs, we update ours to match THREE.
- (:BA)-[:USES]->(:AccessCode) — the code each BA owns and gives to their downline.
- (:BA)-[:SPONSORS]->(:Prospect) — every prospect a BA invited.
- (:Prospect)-[:PLACED_AT]->(:Position) — the holding tank position record.
- Queries answer: “Who is in Paul’s downline?” “How many generations between Kevin and BA X?” “Who sponsored the person who sponsored BA Y?”
### ChromaDB — semantic search
ChromaDB stores embeddings of text content, used for fuzzy lookups and future agent retrieval. Not the system of record — a lookup layer.
- tm_bas — BA identity blobs (name, role, sponsor, location), searchable by free text.
- tm_michael_transcripts — the voice agent interviews, searchable by topic.
- tm_prospect_activity — prospect placement events and callback notes.
- tm_audit_log — system events for retrospective debugging and pattern detection.
## C.4  External services
- Telnyx — SMS to BAs (callback alerts) and outbound voice (powers Michael’s interview calls). Already used in the existing partial build; pattern proven.
- Email provider (TBD) — For welcome emails, webinar confirmations, password resets. Open between Resend, Postmark, SendGrid, or SES. Decision deferred until first email is needed.
- YouTube IFrame API — Hosts Dr. Dan’s 17-minute video. The IFrame API’s “ended” event is what fires video_complete on the .com client.
- THREE International — The final authority and single source of truth for all sponsorship, enrollment, placement, and compensation. Off-app. Sponsorship and enrollment happen there. Team Magnificent does not have an API integration with THREE — the connection is manual, BA-to-BA. The THREE BA ID captured at signup is the link. If our records ever differ from THREE’s, we update ours to match.
## C.5  How a single user action travels through the system
Two end-to-end examples to make the architecture concrete.
### Example 1: A new BA signs up
- User clicks “Create my account” on /register in the .team client.
- Browser POSTs to /api/auth/register with the form payload.
- Server’s Auth module validates the access code, looks up the sponsor BA, validates email and THREE BA ID uniqueness, hashes the password.
- Server’s BA domain module creates the new BA record by calling the persistence gateway.
- Gateway writes to MongoDB (insert bas row), Neo4j (MERGE the new (:BA) node and the [:SPONSORED] edge from sponsor to new BA), and ChromaDB (add identity blob).
- Server’s Notifications module enqueues Michael’s outbound call and sends the welcome email.
- Server returns success + Set-Cookie with JWT.
- Client redirects to /welcome.
### Example 2: A prospect finishes the Dr. Dan video
- YouTube IFrame fires the “ended” event in the .com client.
- Client POSTs to /api/placement/video-complete with the invitation token.
- Server’s Invitations module resolves the token, looks up the sponsor BA and the prospect record (or creates it).
- Server’s Placement module assigns the next sequential position number.
- Gateway writes to MongoDB (update prospect with position), Neo4j (create [:PLACED_AT] edge), ChromaDB (add prospect activity record).
- Server’s Realtime module pushes the new placement to every connected dashboard via SSE.
- Server returns the position number to the prospect’s client.
- Client transitions from presentation page to dashboard, showing position #N.
- Every other prospect currently viewing the dashboard sees a new card appear in their Live Placements section (with name and timestamp).

# D.  Access codes

Access codes are the gate, the genealogy record, and the trust mechanism that keeps Team Magnificent exclusive. Their behavior is precise and intentional.
## D.1  Format
- Pattern: TM-XX where XX is 2 to 4 alphanumeric characters.
- First wave: TM-01 (Kevin), TM-02, TM-03, etc. Sequential.
- Codes are uppercase, hyphenated. Short enough to say over the phone, write on paper, text in a single line.
- Codes contain no PII — they are opaque identifiers.
## D.2  Lifecycle
- Generation — Kevin generates each code from the admin dashboard. No other path — not the .team client, not by API call from sponsors. Always Kevin. Always the admin dashboard.
- Assignment — At generation, Kevin assigns the code to a specific BA. That BA owns the code from that moment forward, for life.
- First use — When a new BA uses the code at signup, the code is not consumed. The system records the new BA as sponsored by the code’s owning BA, and the code remains active.
- Reuse — The same code is used by the same BA for every new BA they ever sponsor. Paul gives TM-07 to every person he brings into Team Magnificent.
- Revocation — Kevin can deactivate a code from the admin dashboard if it is being misused. Already-signed-up BAs keep their sponsor binding — only future signups are blocked.
## D.3  Genealogy — the second confirmation
Every code traces back through the chain to TM-01. The chain is a directed path in Neo4j: KEVIN (TM-01) → PAUL (TM-07) → NEW BA → NEW BA’s downline — each step recorded by the [:SPONSORED] edge created at signup.
This becomes an independent record of who sponsored whom on Team Magnificent. THREE International keeps its own genealogy in its own systems. Team Magnificent keeps its own. If the two ever diverge — data error, dispute, ambiguity — the Team Magnificent record stands as a separate source of truth, anchored to who held the code at signup. This is what Kevin meant when he said “this will be another confirmation of genealogy.”
## D.4  Rules the system enforces
- A code must exist and be active before any signup using it can complete.
- A code’s owning BA is set at generation and cannot change.
- A new BA’s sponsor is determined entirely by the code used at signup. The sponsor field cannot be edited later. Same principle as prospect sponsor immutability, applied to BA-to-BA.
- A BA may hold exactly one access code at any time.
- Codes are not transferable between BAs. If a BA leaves or is removed, their code is deactivated and any future BA they would have sponsored must come in through someone else’s code.
## D.5  What the admin dashboard does with codes
- Generate a new code, assign to a named BA, enter a note. The system shows the generated TM-XX value.
- View every code: code, owning BA, sponsor of owning BA (one level up the chain), status, when generated, count of BAs signed up under it.
- View the full genealogy tree, rooted at TM-01 (Kevin), expandable level by level.
- Compare any BA’s genealogy in the access-code system against their reported sponsor in THREE — catches discrepancies.
- Deactivate a code (does not affect existing signups; only blocks new ones).

# E.  Open questions

Details that need decisions before code is written. They are not assumptions. They are decisions for Kevin.
## E.1  On access codes
- Exact format width: TM-XX (2 chars), TM-XXX (3 chars), or TM-XXXX (4 chars)? Kevin’s example was TM-01 (2-char numeric). At scale, where the team is 100,000 BAs, codes would need to be longer than 2 chars. Confirm the chosen width.
- Alphabet: numeric only (0-9), alphanumeric (0-9 + A-Z), or excluding visually ambiguous characters (no 0/O, no 1/I)?
- How does a BA request their access code from Kevin in practice? Two options confirmed in conversation: (a) inside the .team site — a “Request my access code” button surfaces in their cockpit, you see it pending in admin, you approve and the code is assigned; (b) outside the app entirely — they text/call/message you. Both/either?
## E.2  On the signup form
- Does the BA enter their THREE International BA ID at signup, or can it be filled in later from the BA profile page? The system needs it eventually to confirm genealogy against THREE’s records — but is requiring it at signup a friction point that should be relaxed?
- Email verification before account is fully active — yes or no? If yes, what is the verification email and what does it link to?
- Should the page show the sponsor’s photo when a valid code is entered (“Sponsored by Paul Barrios [photo]”)? Adds warmth and a check on the BA’s end that they got the right code.
- Password requirements — the proposed minimum is 8 characters with at least one letter and one number. Is that the right bar?
## E.3  On the admin dashboard
- First version of admin — what does it absolutely need on day one? Likely: a code generator (form to assign codes to BAs), a code list, a view of new signups.
- Who is in admin besides Kevin? Paul also? Just Kevin? Eventually a small support team?
- When a BA requests their access code via option (a) above, what should the approval action be — one click to approve and auto-generate? A code-entry field? Something else?
## E.4  On the welcome and Michael flow
- What does the welcome screen actually say, in your words?
- Does Michael call immediately on signup, or after a delay (e.g. “Michael will call you within the next 30 minutes” so the BA can put the phone down and answer when it rings)?
- What is Michael’s opening line, his question script, his closing? Is there an existing script document, or does this need to be written?
- If the new BA does not answer Michael’s call, what is the fallback? A second attempt? An email? A SMS link to a written version of the same intake?
## E.5  On the Fast Start Guide
- Who writes the comp plan training, the binary placement explainer, the product training, the initial-list builder? Does this content already exist somewhere (a Google Doc, a PDF, an old website)?
- Is the Fast Start Guide a single long page the BA scrolls through, a set of modules with checkmarks, a sequence of videos, or a mix?
- Does the BA need to complete each step before moving to the next, or can they move freely?
## E.6  On infrastructure
- Email provider: Resend, Postmark, SendGrid, or SES? Decision deferred until first email is sent. Resend tends to be the fastest to integrate.
- Where does the app run in production? Local for now, eventually on a VPS or PaaS? Domain DNS already pointing somewhere?
- Should the existing momentum-creation-system repo be salvaged, archived, or deleted entirely as Team Magnificent moves to a fresh foundation?
